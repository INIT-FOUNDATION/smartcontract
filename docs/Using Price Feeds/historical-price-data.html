However, `AggregatorV3Interface` also exposes functions which can be used to retrieve price data of a previous round ID. There are two parameters that can cause Chainlink nodes to update: | Name | Description | | ------------------------------- | -------------------------------------------------------------------- | | Deviation Threshold | Chainlink nodes are monitoring prices of assets off-chain. The deviation of the real-world price of an asset beyond a certain interval triggers all the nodes to update. | | Heartbeat Threshold | If the price stays within the deviation parameters, it will only trigger an update every *X* minutes / hours. | You can find these parameters at [data.chain.link](https://data.chain.link/) on an example like [ETH / USD](https://data.chain.link/ethereum/mainnet/crypto-usd/eth-usd). To learn more about how price feeds update, see the [Decentralized Data Model](../architecture-decentralized-model/#aggregation-parameters) page. # Historical Rounds To get historical data, you need to call the `getRoundData` [function](/docs/price-feeds-api-reference/#getrounddata) and provide `roundId` as a parameter. Data feeds are updated in **rounds**. Rounds are identified by their `roundId`, which increases with each new round. This increase may not be monotonic. Knowing the `roundId` of a previous round allows contracts to consume historical price data. The `roundId` is computed as the following in the [proxy contract](https://github.com/smartcontractkit/chainlink/blob/develop/contracts/src/v0.7/dev/AggregatorProxy.sol): ```solidity roundId = uint80((uint256(phaseId) << 64) | originalId); ``` where: - `phaseId` is incremented each time the underlying aggregator implementation has been updated. More info on the relationship between _proxies_ and _aggregators_ can be found [here](/docs/architecture-decentralized-model/). - `originalId` is the round id of the current underlying aggregator. The id starts at 1. - `roundId` computed round id. From the above formula, you can think of it as a large number containing `phaseId` and `originalId`. When querying the historical data, it is important to know when we hit the history of the underlying aggregator. Let's illustrate this with an example: At the time of writing, `latestRoundData` [function](/docs/price-feeds-api-reference/#latestrounddata) of [LINK / USD feed on Ethereum Mainnet](/docs/ethereum-addresses/) returns _roundId = 92233720368547771158 _. Using this value, we can compute the `phaseId` and `originalId`: - _phaseId = 92233720368547771158 >> 64_ . Right shifting an integer by 64 bits is equivalent to dividing it by 2^64: _phaseId = 92233720368547771158/ 2^64 = 5_ . The current phase id is 5 , which means that this proxy has had 5 underlying aggregators since its initial deployment. - _originalId = uint64(92233720368547771158)_. This will retrieve the first 64 bits from the right. If you are calculating this off-chain, you can do the following in javascript: ```javascript // First parse to BigInt to perform computation with big integers const num = BigInt("92233720368547771158"); const num2 = BigInt("0xFFFFFFFFFFFFFFFF"); // Largest 64bits integer console.log(Number(num >> 64n)); // returns 5 (phaseId) console.log(Number(num & num2)) // returns 13078 (originalId) . Use & (AND bitwise operator) which sets each bit to 1 if both bits are 1 ``` Now that we have the current aggregator's round _13078_ , we can get its historical data by looping over `getRoundData` [function](/docs/price-feeds-api-reference/#getrounddata) : - starting from the first round 92233720368547758081 (result of _92233720368547771158 - 13078 + 1_). - until the current round: _92233720368547771158_. _Note_: if we want the historical data for previous aggregators, we have to decrement the `phaseId` and start from round 1. For phase 4, we can get the starting roundId off-chain in javascript : ```javascript const phaseId = BigInt("4"); const originalId = BigInt("1"); roundId = (phaseId << 64n) | originalId // returns 73786976294838206465n ``` We can loop over the `getRoundData` [function](/docs/price-feeds-api-reference/#getrounddata) , starting at _73786976294838206465_ and incrementing until we get a revert. This means that we have reached the last round for the underlying aggregator. The same process could be repeated for previous `phaseIds` (3,2,1). A few important words regarding the returned values of the `getRoundData` [function](/docs/price-feeds-api-reference/#getrounddata): The `updatedAt` data feed property is the timestamp of an answered round and `answeredInRound` is the round it was updated in. You can check `answeredInRound` against the current `roundId`. If `answeredInRound` is less than `roundId`, the answer is being carried over. If `answeredInRound` is equal to `roundId`, then the answer is fresh. > ðŸš§ Important > > A read can revert if the caller is requesting details of a round that was invalid or has not yet been answered yet. If you are deriving a round id without having observed it before, the round may or may not be complete. To check if it is, you must validate that the timestamp on that round is not 0. In a best-case scenario rounds update chronologically. However, a round can time out if it doesn't reach consensus. Technically, that is a timed out round that carries over the answer from the previous round. The `roundId` can jump significantly when the `phaseId` is updated because of how the combination of `phaseId` and `roundId` is stored in the proxy. ## Solidity ```solidity Kovan {% include 'samples/PriceFeeds/HistoricalPriceConsumer.sol' %} ```
